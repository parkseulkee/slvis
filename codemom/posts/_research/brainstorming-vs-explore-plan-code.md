# brainstorming × explore-plan-code — 두 스킬 정리

> 목적: 정보 공유 포스트(@codemom.life)의 원천 리서치. 스킬 정의, 내가 실제로 어떻게 쓰는지, `~/app/dailyclip`에서 쌓인 산출물 패턴을 한 문서에 모아둠.
> 작성: 2026-04-23

---

## 한 줄 요약

- **brainstorming** = 코드를 치기 전에 "뭘 만들지"를 합의시키는 스킬. 끝에 `design spec`을 파일로 남기고 커밋한다.
- **explore-plan-code** = "어떻게 만들지"를 합의시키는 스킬. 끝에 **커밋 분할 테이블까지 있는 plan**을 남기고 코딩으로 넘어간다.
- 내 워크플로우 = **의도가 모호하면** brainstorming → (승인) → explore-plan-code. **구현 방향이 명확하면** 바로 explore-plan-code.

## 1. brainstorming 스킬 — 공식 정의

`~/.claude/skills/brainstorming/SKILL.md`

### 역할

> "아이디어를 협업 대화로 굴려서 '설계된 것'으로 만든다. 구현 전에 탐색한다."

### 하드 게이트

스킬 문서 맨 위에 이렇게 박혀 있음:

> Do NOT write code, scaffold projects, invoke implementation skills, or take any implementation action until you have presented a design and the user has explicitly approved it. This applies to ALL projects — no exceptions for "simple" work.

번역: **유저가 디자인을 명시적으로 승인하기 전까지는 단 한 줄도 코드 금지.** "간단한 거라서"는 안 먹힘.

### 순서(체크리스트)

1. 프로젝트 컨텍스트 탐색 (파일·문서·최근 커밋)
2. 비주얼 컴패니언 제안 (시각 논의가 필요해 보이면, 단독 메시지로 한 번)
3. **한 번에 한 질문씩** 명확화 — 목적·제약·성공 기준 위주, 가능하면 객관식
4. **2~3개 접근안 제시** (트레이드오프 + 추천안 + 그 이유)
5. 설계 섹션을 하나씩 보여주고 섹션마다 승인 받기
6. `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 로 스펙 파일 쓰고 커밋
7. 스펙 자가 리뷰 — placeholder·모순·모호함·범위 초과 즉시 수정
8. 유저에게 스펙 리뷰 요청 → 승인 대기
9. **터미널 상태로 `explore-plan-code` 호출** — 다른 구현 스킬 금지

### 핵심 원칙

- **YAGNI 무자비하게**: 모든 디자인에서 불필요한 기능 제거
- **한 번에 한 질문**: 압도하지 않기
- **객관식 선호**: 답하기 쉬움
- **대안 항상 탐색**: 2~3개 비교 후 결정
- **점진 검증**: 제시 → 승인 → 다음

### 산출물 모양 (실제 dailyclip 예시)

`docs/superpowers/specs/2026-04-22-clip-save-original-design.md` 의 섹션 구성:

```
# 클립 원본 저장 기능 설계
## 목적
## UI (위치·아이콘·스타일·상태 표현 테이블)
## 저장 플로우 (번호 스텝)
## 변경 파일 (파일·변경 내용 테이블)
## 권한 문구
## 주의사항
```

"뭘 만들지"에 집중. 커밋 분할이나 리스크 컬럼 같은 **실행** 얘기는 없음.

## 2. explore-plan-code 스킬 — 공식 정의

`~/.claude/skills/explore-plan-code/SKILL.md`
출처 표기: Anthropic best-practices — "Explore first, then plan, then code".

### 역할

> "머리로 바로 코드를 치면 엉뚱한 문제를 푼다. 리서치와 실행을 분리한다."

### 적용 범위

- **모든 코드 변경**. 크기 무관. 기능·리팩터·버그 수정·한 줄 수정·오타까지.
- 사소한 diff는 Explore/Plan을 한두 문장으로 압축 가능. 하지만 **쓰기는 쓴다.**
- 건너뛰는 유일한 경우: 순수 Q&A (코드 변경 없음), 또는 유저가 "그냥 해" 라고 명시했고 내가 "plan 스킵함" 플래그 띄운 경우.

### 네 단계

1. **Explore (Plan Mode)**
   - 파일 읽기·질문 답변만. 수정 금지.
   - 직접 관련 파일 + 주변 패턴(이 코드베이스에서 비슷한 기능이 어떻게 만들어졌는지) 같이 읽기
   - 범위 크면 **서브에이전트에 위임**해서 메인 컨텍스트 오염 방지
   - 재사용 가능한 유틸 먼저 찾기

2. **Plan (Plan Mode 계속)**
   - 변경 파일(경로)
   - 새로 만들 함수·모듈 / 재사용할 것(경로 인용)
   - 데이터 흐름
   - 테스트 전략
   - 리스크·미지수
   - **커밋 분할 테이블 (필수)**

3. **Code (Normal Mode)**
   - plan 기준으로 구현. 재해석 금지.
   - 계획과 현실이 어긋나면 조용히 넘어가지 말고 plan 업데이트.

4. **Commit**
   - plan의 커밋 단위대로. 메시지는 **왜**에 집중.
   - plan과 달리 합치거나 쪼갰으면 그 사실을 명시.

### 커밋 분할 테이블 (필수 포맷)

| #   | 커밋 제목         | 범위       | 왜 (한 줄)       | 리스크 | 의존 |
| --- | ----------------- | ---------- | ---------------- | ------ | ---- |
| 1   | `Add X utility`   | `src/x.ts` | 공유 헬퍼        | Low    | —    |
| 2   | `Wire X into Y`   | `src/y.ts` | 사용자 체감 동작 | Med    | 1    |
| 3   | `Remove legacy Z` | `src/z.ts` | Y가 X 쓰니 정리  | Low    | 2    |

규칙:

- **한 커밋 = 한 논리 변경**. 리팩터 + 기능 = 두 커밋.
- 각 커밋은 **빌드·테스트 통과 상태로 트리를 남긴다**. 아니면 명시.
- 큰 커밋 1개보다 작은 커밋 3~5개 선호. 단 인위적 5분할보다 1개가 낫다.
- 리네임·포맷 같은 **기계적 변경은 별도 커밋**. diff 리뷰성 ↑.

### 안티패턴 (스킬 문서가 직접 경고)

- "머릿속 플랜" — 종이에 안 썼으면 코딩 중에 표류함
- 범위 없는 탐색 — 컨텍스트 다 태워먹음
- 검증 생략
- 두 번 넘게 고쳐쓰기 — 그때는 `/clear` 하고 재시작
- 계획 몰래 수정 — 틀렸으면 "틀렸다"고 말하고 고칠 것

### 다른 스킬과의 관계

- 요구사항 자체가 불분명(유저가 "대시보드 만들어줘"라고만 함) → **먼저 brainstorming**으로 스펙 잡고, 이 스킬로 구현 계획.
- 진짜 사소한 diff는 이 스킬 없이 그냥 한다.

### 산출물 모양 (실제 dailyclip 예시)

`docs/plans/2026-04-23-overlay-font-size-controls.md` 의 섹션 구성:

```
# 내보내기 오버레이 폰트·사이즈 제어 확장
## 배경
## 결정 (폰트 리스트 공유 / 사이즈는 슬라이더 / 세그먼트 전환)
## 데이터 모델 (UserDefaults 키 테이블 + 안전장치)
## 데이터 흐름 (다이어그램)
## 변경 파일 (번호 리스트, 파일별 변경 사항)
## 검증 (빌드·수동 검증 체크리스트)
## 커밋 분할 (3행 테이블)
## YAGNI / 비목표
```

**커밋 분할 테이블은 빠지지 않는다.** 스킬이 강제하는 구조.

## 3. 두 스킬 비교 — 한눈에

| 축               | brainstorming                        | explore-plan-code                      |
| ---------------- | ------------------------------------ | -------------------------------------- |
| 답하는 질문      | **뭘** 만들지                        | **어떻게** 만들지                      |
| 입력             | 흐릿한 아이디어·의도                 | 명확한 요구사항·스펙                   |
| 출력             | 디자인 스펙 (design.md)              | 실행 계획 + 커밋 분할 테이블           |
| 저장 위치        | `docs/superpowers/specs/…-design.md` | `docs/plans/…-design.md`               |
| 대화 스타일      | 한 번에 한 질문, 객관식, 섹션 승인제 | 파일 읽고 plan 작성 → 일괄 승인        |
| 종료 상태        | → explore-plan-code 호출             | → 코드 작성 → 커밋                     |
| "코드 금지" 구간 | 디자인 승인 전까지 절대              | Plan Mode 동안 (탐색·계획은 수정 없이) |
| 테이블 필수 여부 | 없음                                 | **커밋 분할 테이블 필수**              |

## 4. 내가 실제로 쓰는 법 (`~/app/dailyclip`)

### 4-1. 내 프로젝트 CLAUDE.md가 강제하는 룰

`~/app/dailyclip/CLAUDE.md` 워크플로우 섹션:

> 기능 추가·수정·리팩터링 등 **코드 변경 요청이 들어오면 먼저 `brainstorming` 또는 `explore-plan-code` 스킬을 호출**한다. 크기와 무관. 의도·방향 애매하면 `brainstorming`, 구현 방향이 명확하면 `explore-plan-code`.
>
> 작업 단위가 끝나면 빌드를 통과시킨 뒤 **알아서 커밋**한다. 메시지는 한국어 한 줄.
>
> 계획 문서는 한국어로 작성.

즉 **"두 스킬 중 하나를 타지 않고 코드로 내려가지 마라"**를 프로젝트 규약으로 박아뒀음.

### 4-2. 스킬 선택 기준 (실제 판단)

| 요청 유형                                                                             | 선택              | 이유                                          |
| ------------------------------------------------------------------------------------- | ----------------- | --------------------------------------------- |
| "클립 원본을 카메라 롤에 저장하는 버튼 만들어줘"                                      | explore-plan-code | 위치·동작 명확, 바로 plan 가능                |
| "앱 톤이 밋밋한 거 같은데 감성 브이로그 느낌으로 전환하고 싶어"                       | brainstorming     | "어떻게"가 아니라 "뭘"부터 합의 필요          |
| "REC 버튼을 pill에서 원형 셔터로 바꿔"                                                | brainstorming     | 시각 결정 다수 (크기·그라데이션·Press 스타일) |
| "CaptionFontFamily를 OverlayFontFamily로 리네이밍하고 시간/캡션 사이즈 슬라이더 추가" | explore-plan-code | 리네이밍 + UI 붙이기, 구현 스펙 이미 결정됨   |
| "홈에서 REC 버튼 원형 물리감 구현"                                                    | explore-plan-code | 시각 결정을 사전에 brainstorming으로 끝낸 후  |

**패턴 감각**: "결정이 필요한가 vs. 결정은 끝났고 실행만 남았나"

### 4-4. 실제 파이프라인 전형 — "감성 브이로그 전환" 사례

1. 유저(나): "앱 톤이 어긋나. 감성 브이로그로 가고 싶어" (의도 모호)
2. → `brainstorming` 호출
3. 스킬: 현재 컬러 팔레트·폰트 구조 탐색 → 2~3안 제시 → 섹션별 승인
4. 산출: `docs/superpowers/specs/2026-04-15-design-overhaul-design.md`
5. 유저 승인
6. 스킬이 자동으로 `explore-plan-code` 호출
7. 스킬: 리스크 낮은 순서로 **Phase 1~N** 분할, 각 Phase = 1 커밋
8. 산출: `docs/plans/design-soft-vlog.md` (여백 → 세리프 → 시맨틱 토큰 → 라이트 팔레트 순)
9. 코드 작성 + 커밋 (Phase마다)
10. 빌드 검증 → 한국어 한 줄 커밋 메시지 자동 작성

이전 시도에서 한 번에 테마·컬러·배경을 다 뒤집다가 **검은 화면 회귀 + 수동 복구 혼선**이 있었던 걸 plan 문서 배경에 명시. plan 덕분에 롤백 지점이 명확해짐.

### 4-5. 단축키가 있다

```
의도·방향 애매            → brainstorming
                              ↓ (스펙 승인)
                              ↓ (스킬이 자동 호출)
구현 방향 명확하게 떨어짐 → explore-plan-code
                              ↓ (plan 승인)
                            코드 + 커밋
```

중간을 건너뛸 수 있음. 명확하면 바로 오른쪽으로. **근데 "명확함"을 내가 과신하면 함정.** 헷갈리면 왼쪽부터.

## 5. 왜 이 두 스킬을 쓰는가 (정보 공유 포스트용 메시지 후보)

포스트에 쓸 만한 앵글들:

### A. "코드 못 치는 구간을 만드는 게 오히려 빠르다"

- 두 스킬의 공통점 = **코드 금지 구간을 강제로 낑겨 넣는다**
- brainstorming: 디자인 승인 전까지 코드 금지
- explore-plan-code: Plan Mode 중엔 수정 금지
- 역설: 금지 시간이 길수록 결과가 빨라짐. "엉뚱한 문제를 안 풀기" 때문.

### B. "커밋 분할을 AI가 대신 계획해주면 벌어지는 일"

- 커밋 분할 테이블 = 리뷰어 친화 + 롤백 포인트 + 리스크 가시화
- 혼자 개발할 때도 **내일의 나를 위한 리뷰어**
- 실제 감성 브이로그 전환에서 이 덕분에 단계적 복구 가능

### C. "의도 모호 vs 방향 명확 — 스킬 두 개로 가르는 이유"

- 실수 유형 1: 모호한데 바로 explore-plan-code → 엉뚱한 걸 멋지게 만듦
- 실수 유형 2: 명확한데 brainstorming → 회의만 하다 시간 날림
- 해결: 진입 때 한 번만 판별하면 나머지는 레일 위에서 굴러감

### D. "CLAUDE.md에 워크플로우를 박아두면 잔소리가 줄어든다"

- `~/app/dailyclip/CLAUDE.md` 하나에 "코드 요청 오면 둘 중 하나 호출"을 명시
- 매번 "먼저 계획 세워줘"라고 타이핑할 필요 없음
- 스킬과 프로젝트 메모리의 콤보 = 자동화된 엔지니어링 태도

### E. "실제로 남는 것" — 9일 13개 문서

- brainstorming 8 + plan 5
- 코드와 함께 리포에 커밋되므로 git history로 **의사결정 타임라인** 구축
- 2026-04-18 썸네일 스펙처럼 **사후에도 현황 문서로 재활용** 가능

- 존댓말 캐주얼 (~했어요 / ~해봤어요)
- "PLOT TWIST" 같은 자기선언 라벨 금지
- 슬라이드 폰트 최소 24px, 악센트 컬러 포스트당 하나
- 카피 톤: "이거 써봤는데 이래서 좋았어요" 정물화 톤. 훅은 "코드 AI가 코드를 못 치게 하는 구간이 제일 빠르더라" 같은 역설 방향이 어울림.
- 스킬 이름은 코드 폰트(JetBrains Mono) 라벨로 처리하기 좋음

## 7. 덧붙이면 좋은 디테일

- `brainstorming` 스킬 안에 **visual-companion** 기능이 있음. 브라우저로 mockup·다이어그램 띄움. 토큰 많이 먹어서 상시는 아니고 시각 논의 구간에만.
- `brainstorming`은 한 번 호출되면 **터미널 상태가 explore-plan-code**. 다른 구현 스킬(예: 자체 refactor 스킬 같은 거) 호출 금지.
- `explore-plan-code`는 Anthropic best-practices 문서(code.claude.com/docs/en/best-practices)의 "Explore first, then plan, then code"를 그대로 스킬화한 것. 즉 **공식 가이드 그 자체**.
- 두 스킬 모두 **서브에이전트 위임**을 권장. 탐색으로 메인 컨텍스트 태우지 말 것.
- plan 문서 섹션 관례(내 프로젝트 기준): `배경 → 결정 → 데이터 모델 → 데이터 흐름 → 변경 파일 → 검증 → 커밋 분할 → YAGNI`
